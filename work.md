🛠️ 各阶段深度拆解与多智能体交互
阶段一：任务捕获与前置处理（主终端 Master）
用户行为： 乘客在车辆行驶时，看到窗外有感兴趣的建筑，在智能车窗上点击或注视了某个区域。

主终端响应： 主终端获取到点击的物理坐标 (X,Y) 以及当前车载摄像头拍到的全景街景图片。

任务切片： 主终端算法以坐标为中心，切出 3 个可能包含地标建筑的局部图像块（图像 A、图像 B、图像 C），准备分发。

📢 看板实时刷新： [System] 捕获到车窗点击事件(X:142, Y:582)，全景图切片完成，生成 3 个子任务。

阶段二：相同角色并行执行（满足要求 1）
多代理派生： 主终端利用 opencode 同时拉起 3 个完全相同的 Vision Agent（视觉识别智能体）。

高并发执行： 3 个 Agent 共享相同的 VLM（视觉语言模型）工具链，但在多线程下并行处理不同的图片：

Vision Agent 1 负责处理图像 A（识别出是一座中式古建筑）。

Vision Agent 2 负责处理图像 B（识别出是一家知名百年餐饮老店）。

Vision Agent 3 负责处理图像 C（识别出是一栋现代写字楼）。

结果汇总： 并行结束后，3 个 Agent 将识别到的标准实体名称（如：景山万春亭、全聚德烤鸭店、联想总部大楼）返回给主终端。

📢 看板实时刷新： > ● Vision_Agent_1 : RUNNING | ● Vision_Agent_2 : RUNNING | ● Vision_Agent_3 : RUNNING
[Parallel 日志] Vision_Agent_1 识别成功：【景山万春亭】
[Parallel 日志] Vision_Agent_2 识别成功：【全聚德烤鸭店】

阶段三：不同角色之间相互协作与交接（满足要求 2）
当名字识别出来后，单纯的“名字”对乘客来说不够有趣。主终端开始指挥不同角色的 Agent 进场打配合。

交接环节点 1：视觉 ➡️ 检索（RAG）
动作： 主终端将 Vision Agent 识别出的建筑名字组装成结构化的 JSON 状态文件，传递给 RAG Search Agent（知识检索智能体）。

协作： RAG Agent 收到名字后，调用本地的知识库工具（比如你的食品项目知识库、或者旅游百科数据库），检索这些地标的深度信息。

对于“万春亭”，检索其建造年份、历史典故。

对于“全聚德”，检索其招牌菜品、食品安全评级、特色工艺。

📢 看板实时刷新： [Handover] 视觉团队 -> RAG 检索团队。当前执行：RAG_Agent 正在调取【全聚德】的食品非物质文化遗产数据...

交接环节点 2：检索 ➡️ 编剧（Writer）
动作： RAG Agent 将检索到的一大堆硬核、零散的文本资料打包，交接给 Copilot Writer Agent（文案润色智能体）。

协作： Writer Agent 负责把干瘪的数据转换成温和、亲切的车载导游播报词。

生成文案： “乘客您好，您右手边正在经过的是万春亭。它是北京中轴线上的最高点。顺便提一句，前方不远处的全聚德烤鸭店是著名的百年老字号，它们的挂炉烤鸭技术非常地道，推荐您午餐去尝尝……”

阶段四：安全质检与合规性审查（质检协作）
交接环节点 3：编剧 ➡️ 红队质检（Critic）
动作： 润色好的播报文案在推送给用户前，必须经过最后一关。文案被移交给 Critic Verifier Agent（合规质检智能体）。

协作： Critic Agent 负责检查文案中是否存在大模型的“幻觉”（胡编乱造的历史）、是否有不合规的商业广告词。

如果检查通过，向主终端发送 PASSED。

如果发现全聚德的地址写错了，立刻退回给 Writer Agent 重写（形成局部双向协作循环）。

阶段五：成果展示与状态终结（满足要求 3）
最终交接： 质检通过后，最终的结构化导游文本和切片高亮图像被交还给主终端。

车载渲染： 系统调用车载 HMI 接口，在车窗透明屏上高亮圈出那几个建筑物，弹出精致的图文卡片，并通过车载音响进行 TTS 语音播报。

完成状态： 终端面板上的总体进度条冲到 100%，所有 Agent 回归 IDLE（休眠）状态。

📢 看板实时刷新： [System] 最终导游文本通过校验，已成功推送至车载大屏。任务圆满完成！
